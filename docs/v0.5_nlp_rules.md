# 📋 Pyqttyai v0.5 — NLP Rules Engine Guide

> Companion to the [v0.5 README section](../README.md#v05--pyqttyai-v05--voice--nlp-intelligence-layer).
> Covers the **deterministic natural-language-to-command** layer: how rules
> work, how to write them, how to test them, and how to use
> **regex101.com** as an external playground.

Open with: **Settings → 📋 NLP Rules Editor…** or `Ctrl+Shift+R`.

---

## 🎯 Why a Rules Engine (and Not an LLM)?

A short, opinionated answer:

| Concern | LLM | Rules engine |
|---------|:---:|:------------:|
| Latency | 🐢 100–30 000 ms | ⚡ <1 ms |
| Determinism | ❌ Probabilistic | ✅ Same input → same output |
| Hallucinations | 😱 Possible | 🚫 Impossible |
| Offline | ⚠️ Heavy local model | ✅ Pure Python |
| Resource cost | 🔥 GB of RAM | 🪶 Negligible |
| Debuggability | 🌫️ Black box | 🔍 Every step inspectable |

For **real-time voice commands** ("save all configs", "hostname router
twenty"), determinism wins. v0.5 reserves probabilistic agentic AI for
**v0.6 (Pyqttyto)**, which will sit *on top of* this rules layer.

## 🧩 Mental Model

Each **rule** has four ingredients:

```
┌────────────────────────────────────────────────────────────┐
│  🧩 Fragments           — named regex pieces (reusable)    │
│       host = \S+                                           │
│       pwd  = \S+                                           │
│                                                            │
│  🎯 Pattern Template    — how fragments fit together       │
│       name\s+{host}.*password\s+{pwd}                      │
│                                                            │
│  🔧 Handler             — what to do when it matches       │
│       compose_replace                                      │
│                                                            │
│  🔧 Replacements        — output template + transforms     │
│       hostname \U1\nenable secret @\2!\n                   │
└────────────────────────────────────────────────────────────┘
```

For input:
```
the name router1 has password firefox
```
the rule produces:
```
hostname ROUTER1
enable secret @firefox!
```

## 🗂️ The Visual Editor (`Ctrl+Shift+R`)

The Rules Editor is a four-pane workbench:

```
┌──────────────────┬────────────────────────────────────────────┐
│ 📋 Rules list    │ 🧩 Fragments                               │
│  [✓] hostname    │   name = \S+                               │
│  [✓] bgp_setup   │   pwd  = \S+                               │
│  [✓] ipv6_addr   │ ────────────────────────────────────────── │
│  [✓] save_all    │ 🎯 Pattern & Handler                       │
│  ➕ ➖ ⎘ ↑ ↓     │   Template: name\s+{name}.*password\s+{pwd}│
├──────────────────┤   Handler:  compose_replace                │
│ 🧪 Tests grid    │   Post:     (none)                         │
│ ✅ name router1… │ ────────────────────────────────────────── │
│ ❌ name x has p… │ 🔧 Replacements                            │
│ ▶ Run All        │   fragment   name   →   \U1                │
├──────────────────┴────────────────────────────────────────────┤
│ 🧪 Quick Test                                                 │
│  [Input              ] → [Output            ] = [Expected   ] │
│  🎙️                                                           │
└───────────────────────────────────────────────────────────────┘
```

### 📋 Rules list (top-left)

- ✅ Checkbox enables/disables a rule
- ✏️ Double-click to rename
- ➕ ➖ ⎘ ↑ ↓ — add / delete / duplicate / reorder
- 🔄 **Order matters** — rules are applied **top-to-bottom**

### 🧪 Tests grid (bottom-left)

Each rule carries its own **test scenarios** (input + expected output):

- 🟢 Green row = ✅ passes
- 🔴 Red row = ❌ fails
- ⚡ **Live recomputation** — every edit re-runs all tests instantly
- ▶ **Run All** opens a detailed Markdown report with per-test diffs
- ⬆ **Upload** promotes Quick Test content to a permanent test

### 🧩 Fragments table (top-right)

Named regex pieces you can reuse. Keeps patterns **readable** and
**maintainable**.

### 🎯 Pattern & Handler (middle-right)

- **Template** — the master regex. Use `{fragment_name}` to embed fragments.
- **Handler** — currently `compose_replace`. Future handlers pluggable via
  `_HANDLERS` in `rules_engine.py`.
- **Post-process** — apply `upper` / `lower` / `title` to the final output.

### 🔧 Replacements table (bottom-right)

A list of `find → replace` steps applied **inside** the matched span:

| Kind | Find | With |
|------|------|------|
| `fragment` | `name` | `\U1` *(uppercase group 1)* |
| `pattern` | `\.` | `_` |
| `none` | *(skip)* | *(skip)* |

### 🧪 Quick Test (bottom)

A scratch area to try **arbitrary input** against the **selected rule**:

- 🎙️ Built-in mic button so you can speak the test phrase
- ⌨️ `Ctrl+Enter` re-runs the test
- 🎨 Color feedback: 🟢 actual == expected, 🔴 mismatch, ⚫ neutral
- ▶ **Run Rules** runs the *entire ruleset* (not just one rule)

## 🔠 Regex Refresher (Just Enough to Start)

If you've never written regex, here's the absolute minimum:

| Pattern | Matches |
|---------|---------|
| `abc` | literal `abc` |
| `\d` | a digit (`0–9`) |
| `\w` | a word character (`a–z`, `A–Z`, `0–9`, `_`) |
| `\s` | whitespace (space, tab, newline) |
| `\S` | non-whitespace |
| `.` | any character |
| `*` | 0 or more |
| `+` | 1 or more |
| `?` | 0 or 1 |
| `[abc]` | one of `a`, `b`, `c` |
| `[^abc]` | none of `a`, `b`, `c` |
| `(...)` | capture group (numbered: `\1`, `\2`, …) |
| `(?P<name>...)` | named capture group (`\g<name>`) |
| `^` / `$` | start / end of line |
| `\b` | word boundary |
| `a\|b` | `a` or `b` |

### Flags

In a Pyqttyai rule, common flags are enabled by default (case-insensitive,
multiline). Use `flags=["i", "m", "s"]` on the rule for explicit control.

## 🌐 Use **regex101.com** as Your Playground

Pyqttyai's rules use **Python's `re` syntax**. The single best external
tool for learning/debugging regex is **[regex101.com](https://regex101.com/)**.

### Setup once

1. 🌐 Open https://regex101.com/
2. 🐍 In the left sidebar, select **Python** as the flavor
3. ✍️ Paste your pattern (with fragments **expanded** — regex101 doesn't
   know about Pyqttyai's `{fragment}` syntax)
4. 📝 Paste a test input in the "Test String" area
5. 👀 The right panel explains **every token** and highlights matches
6. 🎯 Click any match to see all captured groups

### 🔍 Example walkthrough

You write in Pyqttyai:

```
Template:  name\s+{host}.*password\s+{pwd}
Fragments: host=\S+    pwd=\S+
```

To debug on regex101:

1. Paste the **expanded** pattern:
   ```
   name\s+(\S+).*password\s+(\S+)
   ```
2. Paste test input:
   ```
   the name router1 has password firefox
   ```
3. ✅ regex101 highlights `name router1 has password firefox`, shows
   group 1 = `router1`, group 2 = `firefox`, and explains every token.

### Why regex101 rocks

- ✅ Real-time match highlighting
- ✅ Group capture visualization (hover to see what `\1`, `\2` captured)
- ✅ Built-in **regex explainer** ("Explanation" tab)
- ✅ **Quick reference** of all Python regex features
- ✅ Save patterns with a permalink — share with teammates
- ✅ Detects common mistakes (greedy vs lazy, unescaped special chars)
- ✅ **Code Generator** that produces ready-to-paste Python code

### Pro tips for Python regex

| Need | Use |
|---|---|
| Case-insensitive | `(?i)` at the start, or `i` flag in rule flags |
| Multiline | `(?m)` or `m` flag |
| Non-greedy match | `.*?` instead of `.*` |
| Named groups | `(?P<name>…)` — reference as `\g<name>` |
| Optional whole block | `(?:foo)?` |
| Word boundary | `\b` — crucial for keyword matching |
| Multiple alternatives | `(hostname\|nome do host\|nome)` |

## ✨ Backreferences (the Pyqttyai Extras)

In the **"With"** column of a Replacement (or in the Find & Replace bar's
regex mode), you can use:

| Syntax | Effect |
|--------|--------|
| `\1`, `\2`, …, `\12` | Numbered group |
| `\g<name>` | Named group |
| `\U1` / `\U<name>` | 🔠 **UPPERCASE** the group |
| `\L1` / `\L<name>` | 🔡 **lowercase** the group |
| `\T1` / `\T<name>` | 🔤 **Title Case** the group |
| `\n` `\t` `\r` `\\` | Standard escapes |

### 💡 Example

```
Pattern: name\s+(\S+).*password\s+(\S+)
With:    hostname \U1\nenable secret @\2!
```

Input:
```
the name router1 has password firefox
```

Output:
```
hostname ROUTER1
enable secret @firefox!
```

## 🛠️ Authoring Your First Rule — Worked Example

Goal from the demo: speak *"router BGP 1919"* and have it produce a
properly-formatted Cisco BGP config line.

### Step 1 — Create the rule

1. Open Rules Editor (`Ctrl+Shift+R`)
2. ➕ → name it `bgp_setup`
3. Check **enabled**

### Step 2 — Fragments

Add one fragment:

| Name | Pattern |
|------|---------|
| `as_num` | `\d+` |

### Step 3 — Pattern

In **Pattern & Handler**:

```
Template:  router\s+bgp\s+{as_num}
Handler:   compose_replace
Post:      (leave empty)
```

### Step 4 — Replacements

Add one replacement:

| Kind | Find | With |
|------|------|------|
| `pattern` | `.*` | `router bgp \1` |

### Step 5 — Tests

Add three tests:

| Input | Expected |
|-------|----------|
| `router bgp 1919` | `router bgp 1919` |
| `Router BGP 65000` | `router bgp 65000` |
| `router BGP one nine one nine` | ❌ (won't match — digits expected) |

The third test reveals an issue! Add another rule **before** this one that
converts spoken numbers to digits (or rely on Whisper to do it).

### Step 6 — Verify with Quick Test

Speak *"router BGP 1919"* into the Quick Test mic. Output should be
`router bgp 1919`. ✅

### Step 7 — Save

Hit **Save**. The new rule is now applied to every voice transcription
and every `Ctrl+Shift+A` press.

## 🎯 IPv6 Rule (from the demo)

The demo showed a rule triggered by the **keyword "IPv6"**: when present,
the engine formats the subsequent numbers/letters as an IPv6 address.

### Behavior

| Spoken phrase | Output |
|---------------|--------|
| *"IPv6 192.168.10 acad cafe 2 2 3"* | `2001:db8:acad:cafe::2:2:3` |
| *"192.168.10 acad cafe 2 2 3"* (no keyword) | `192.168.10.acad.cafe.2.2.3` (raw) |

The keyword acts as a **routing switch**: with `IPv6`, the rule applies
IPv6 separator logic; without it, the raw text is left alone (and you can
still fix it with `Ctrl+;` in the editor — see [Editor guide](v0.5_editor.md)).

### How the rule looks (simplified)

```
Fragments:
  ipv6_hextet = [0-9a-f]{1,4}

Template:
  \bIPv6\b\s+([0-9a-f.\s]+)

Replacements:
  pattern  →  \1   →  (post-process splits by . / space, joins with :)
```

> 💡 **Real-world tip from the demo:** Whisper sometimes hallucinates extra
> digits (e.g., `16.16.16.16` → `16.16.16.16.16`). Use `Ctrl+Shift+F` to
> remove the offending repeat, or `Ctrl+Shift+A` to re-run the rule after
> a quick manual fix.

## 🧱 Best Practices

- 🧩 **Reuse fragments.** If five rules all need to capture an IP, define
  `ip = \d{1,3}(\.\d{1,3}){3}` once and reference `{ip}` everywhere.
- 🎯 **Anchor when you can.** `\bsave\b` won't match inside `unsaved`.
- 🧪 **Write the failing test first.** Add input → see ❌ → tune pattern →
  see ✅. Same red/green/refactor loop as TDD.
- 🚦 **Order matters.** Put specific rules **before** generic ones.
- 🛡️ **Never trust silence-only audio.** With VAD on (see
  [Whisper Playground](v0.5_whisper_playground.md)), pure silence never
  reaches the rules engine.
- 💾 **Export and version-control your rules.** They live in your user
  config; copy the JSON into git to share with a team.

## 🩹 Troubleshooting

### *"My rule never matches"*

1. Open Rules Editor → select the rule
2. The **Template** field turns 🟥 with a tooltip if your regex is invalid
3. Otherwise, copy the **expanded** template to regex101.com (Python flavor)
4. Paste the offending input
5. Walk through the highlighted matches token-by-token

### *"It matches but the output is wrong"*

- Add the failing input to the **Tests grid** with the expected output
- Click ▶ **Run All** for a detailed diff report
- Backreferences off-by-one? Group numbering starts at **1**, not 0
  (group 0 is the full match)

### *"My rule fires when I don't want it to"*

- Tighten anchors: add `\b` word boundaries or `^`/`$` line anchors
- Move the rule **below** more specific ones
- Disable it (`Ctrl+Shift+R` → uncheck) and confirm the misbehavior
  disappears

### *"Rule throws an error"*

- The status bar shows `⚠ Rule error: …` for 10 seconds
- The raw (unmodified) transcription is injected so you don't lose data
- Fix the rule (typically a bad backreference number) and try again

## 🚀 Where This Goes Next — v0.6 (Pyqttyto)

The rules engine is the **deterministic foundation** that v0.6 will build on:

- 🤖 An **agentic LLM** will *propose* new rules from natural conversation
- 📚 You'll review, edit, and accept them in this same Rules Editor
- 🎯 The runtime stays deterministic — the LLM only writes the *rules*, it
  never *runs* them in the hot path

This separation — **probabilistic authoring + deterministic execution** —
is the central design bet of the project.
